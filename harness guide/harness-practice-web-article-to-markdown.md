# Harness 视角下的优质实践：从网页拉取到可消费 Markdown

本文把一次具体工程行为——**下载掘金文章 HTML、抽取正文、输出含图片的 Markdown**——抽象成可复用的 **Harness 工程** 实践说明。概念对齐自《Harness 工程：不是新词，而是 Agent 工程终于被讲明白了》一文中的判断：**Agent 的核心可以很简单，真正决定成败的是运行环境是否把任务、工具、状态、反馈与验证接好**。

---

## 1. 实践摘要（我们实际做了什么）

| 步骤 | 行为 | Harness 含义 |
|------|------|----------------|
| 1 | 用可脚本化方式拉取指定 URL 的完整 HTML 并落盘 | **持久化中间产物**，便于复跑与审计 |
| 2 | 首次因网络沙箱/代理失败，换到有权限的环境重试 | **错误分类 + 环境能力声明**（网络是否可用） |
| 3 | 用稳定 DOM 选择器定位正文容器，而非整页当正文 | **上下文裁剪**：只把「任务相关」喂给下游 |
| 4 | 移除 `style` / `script`，再转 Markdown | **噪声治理**，降低下游解析与 diff 成本 |
| 5 | 规范化图片 `src`（如 `&#x26;` → `&`） | **协议/序列化细节修复**，避免「看得见、打不开」的假链接 |
| 6 | 依赖安装在项目内 venv，避免污染用户全局 Python | **隔离与可复现**，符合「环境即契约」 |
| 7 | 从页面 `h1` 取标题写入 Markdown 头部 | **任务输出结构化**，人读与机器读都一致 |
| 8 | 检查输出文件大小与正文中的图片语法 | **轻量验收**：证明管道末端真的产出预期形态 |

这不是「模型写得好」的问题，而是**把抓取、解析、转换、验收放在同一套环境里连续做完**的问题——正是文中所说的 **Harness = 让 Agent（或自动化）能被稳定驾驭的环境系统**。

---

## 2. 对照文中的 Harness 八要素

原文归纳的「工程环境总和」至少包括：任务表达、上下文组织、工具治理、状态保存与裁剪、反馈回流、错误分类与重试、安全边界、**验证是否真做好**。下面逐条对应本次实践。

### 2.1 任务如何被表达

- **输入契约**：单一 URL + 明确交付物（「正文 Markdown，且保留图片」）。
- **输出契约**：一个 `.md` 文件，标题与正文分离，图片为 Markdown 图片语法而非仅内嵌 HTML。

这样任何 Agent 或 CI 接手时，**成功/失败判据清晰**，不会滑成「好像抓到了一点网页」。

### 2.2 上下文如何被组织

- **只抽取** `#article-root .article-viewer.markdown-body.result` 内的 DOM，等价于对模型/流水线只做 **Context 裁剪**：导航栏、侧栏、脚本、统计与骨架屏不进正文通道。
- 与文中 **Context Engineering 是 Harness 的一部分** 一致：先决定「给下游什么」，再谈措辞或格式。

### 2.3 工具如何被暴露、治理、拦截

- **工具选型**：`curl` 负责传输边界；Python + HTML 解析 + `markdownify` 负责语义边界。职责分离，便于替换实现（例如以后把 `curl` 换成带 Cookie 的客户端）而不改解析契约。
- **治理**：依赖固定在项目 `.venv`，避免「本机全局装包失败 / 权限拒绝」这类环境噪声混入业务逻辑——对应文中「工具越来越多要有治理」。

### 2.4 状态如何被保存、恢复、裁剪

- **保存**：原始 `.html` 保留，Markdown 为派生产物。长任务或争议时可回到 HTML 复现解析，对应 **checkpoint / 可恢复**。
- **裁剪**：去样式脚本、只取正文子树，避免把整页 DOM 当状态塞进后续步骤。

### 2.5 反馈如何回流

- 管道末端用 **文件大小、行数、是否出现 `![]()`** 等做快速反馈；若正文为空或图片数为 0，应立即回到「选择器是否失效 / 页面是否改版」这一层修正，而不是在 Markdown 层硬编。

### 2.6 错误如何被分类、重试、升级

- **网络类**：沙箱限制、CONNECT 403 → 归类为环境能力问题，**升级**到有外网权限的执行环境，而不是反复调 URL。
- **依赖类**：用户级 `pip install` 权限拒绝 → **升级**为项目内 venv，把「安装」也纳入仓库可复现 harness，而非假设本机已装好一切。

### 2.7 安全边界如何被建立

- 写入范围限定在**工作区目录**；网络按需开启，避免默认全开。
- 不执行页面内脚本、不把不可信 HTML 直接 `eval`；只做结构化解析与转写，降低供应链与 XSS 类风险面。

### 2.8 系统如何验证「真的把事做好了」

- **结构验收**：Markdown 含一级标题、正文段落、列表/引用等是否与源 DOM 一致抽样核对。
- **资源验收**：至少统计图片 Markdown 条数是否与 HTML 中正文图片一致（本次正文为单图场景，亦适用多图扩展）。

文中强调 **「环境够不够好」比「模型够不够聪明」更决定结果**；对纯流水线而言，**验收与可复现环境就是那套「环境」**。

## 3. 可复用的「迷你 Harness」清单

若要把本实践推广到「任意文章站 → Markdown」，建议固定下列检查项：

1. **意图**：URL、是否含登录态、输出路径、是否下载图片到本地。
2. **快照**：始终保留原始 HTML（或 HTTP 响应头中的关键信息），便于站点改版后对照。
3. **选择器策略**：优先 SSR 稳定节点；准备降级（如 JSON-LD、`__NUXT__`）并写进同一仓库文档或代码注释。
4. **规范化**：URL 解码、HTML 实体、`//` 协议相对 URL 补全为 `https:`。
5. **隔离**：语言与依赖版本锁在 venv / 容器；CI 与本地一致。
6. **验收**：最小自动化（行数、关键词、图片链接数）；重要稿件再加人工 spot check。

## 4. 与原文一句话对齐

> **Agent = Loop(LLM + Context + Tools)** —— loop 可以很短；**Harness** 回答的是：**这个 loop 放在什么环境里，才能不在生产里烂掉**。

本次实践没有调用大模型，但同一套逻辑成立：**抓取—解析—转换—验收** 就是一个确定性 loop；**Harness** 体现在 URL 契约、DOM 边界、依赖隔离、失败重试与落盘证据上。把这套做稳，再把 LLM 接进来做摘要、打标签或二次改写时，才不会把「环境欠账」全部推给模型。
